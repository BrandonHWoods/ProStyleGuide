Style Guide


Introduction

This is more than a simple style guide. While it does define styling and usability guidelines for What Counts, it is also a library of examples. Developers can review the code here to find examples of how to implement HTML elements and page elements (widgets) in their presentation code in a way that maintains a consistent look, feel, and user experience across the What Counts platform. All the code on the Style Guide conforms to the standards defined by the Style Guide. Therefore, it is also a practical library of code widgets for copying and modifying—which is encouraged, as it will help facilitate markup consistency.

This is not an exhaustive style guide. It does not stipulate the conditions necessary to create a pixel-perfect website. Rather, it is intended to help create consistency in look and use; therefore, pixel dimensions and red-lines have not been included, as they exceed the intent and scope of the guide.

The majority of the code on the guide can be reviewed for duplication by viewing the source of a given element. There are exceptions to this, notably:

  • - select dropdowns
  • - checkboxes
  • - radio buttons

Flat UI, our user interface framework, overrides the default presentation of these elements with additional HTML markup. HTML Elements that are redrawn by the framework have code examples associated with them that show the correct way to code the elements so that the UI framework will appropriately restyle them.

Flat UI

Flat UI—utilizing Bootstrap, which utilizes both Bootstrap-specific JS, and jQuery-UI—is our front-end framework. Flat UI provides the majority of the CSS styling for the site, and a library of standard functionality—such as the JS for modal dialog boxes, and rendering re-styled HTML elements such as checkboxes. See Bootstrap for more information.

Fonts

Large, bold text—declining to normal, non-bold text—should be used to establish a clear information hierarchy. All titles or sub titles should be marked with their appropriate heading tag (h1, h2, &c.). All pages should, at the least, employ a Page Title and contain a content body that contains either a paragraph, graphic, or table. No special text formatting is required, or allowed, beyond that provided by default. If all text is appropriately marked with the semantically correct HTML tags, the correct formatting will be applied.

All text on the site should conform to the font-family hierarchy:

     "Lato",Helvetica,Arial,sans-serif;

Primary Navigation: The Navigation Menu

The navigation Menu is the only valid place where the What Counts logo may be placed. It must have an id of "NavMenu". It then contains an Unordered List. Each main menu topic that contains sub-content must have a class of "has-sub". Each sub-item that contains flyout content, must contain a class "set-sub toggle-nav".

Selects, Checkboxes, and Radio Buttons

Selects, checkboxes, and radio buttons are implemented using a standard HTML tags; however, they are automatically converted at display time by the system. Select boxes do not require meta tags to correctly render. Checkboxes and radio buttons require a label, and a data attribute to correctly render. The label must have a class of the input type (i.e., checkbox, or radio), and the input must have a "data-toggle" attribute, also of the input type (i.e., checkbox, or radio). Examples are below:

Checkboxes

Using Flat-style checkboxes requires a JS plugin that creates all custom markup. If you correctly code the HTML markup for a checkbox, however, you will not need to initialize them in code. The system will do that for you. Below are some examples of checkboxes:

Usage via data attributes:
<label class="checkbox" for="checkbox1">
  <input type="checkbox" value="" id="checkbox1" data-toggle="checkbox">
  Checkbox
</label>
Methods

.checkbox('toggle')
toggles checkbox state from checked to unchecked and back.

.checkbox('check')
sets checkbox state to checked.

.checkbox('uncheck')
sets checkbox state to unchecked. 

$(':checkbox').checkbox('check');
Events

.on('toggle')
Fired when checkbox state changed between check ? uncheck.

.on('change')
Same as toggle. 

$(':checkbox').on('toggle', function() {
  // Do something
});

Radio buttons

Rendering radio buttons also requires using a JS plugin. Again, if you format the HTML correctly, the system will appropriately render the radio buttons without additional coding.

Usage via data attributes:
<label class="radio">
  <input type="radio" name="group1" value="1" data-toggle="radio">
  Radio is off
</label>

<label class="radio">
  <input type="radio" name="group1" value="2" data-toggle="radio" checked>
  Radio is on
</label>
Methods

.radio('check')
sets radio state to checked.

.radio('uncheck')
sets radio state to unchecked. 

$(':radio').radio('check');
Events

.on('toggle')
Fired when radio state changed between check ? uncheck.

.on('change')
Same as toggle with just one difference — event sends for each radio which state was changed. 

$(':radio').on('toggle', function() {
  // Do something
});

Buttons

Buttons should be used to indicate to a user that an action is possible. Actions are verbs, such as:

Actions are actions from the user's perspective, not the application's. Therefore, there are times when it is appropriate to use a button to navigate to a page—as in a Cancel operation, when the application may redirect the user from one page to another.

Links should never be used for actions. Links should be used to create connections between pages. See Links for more information.

Button action names should be descriptive. The name should clearly describe the action. Prefer clarity to brevity, for example—prefer "Add Subscriber" to "Add".

Disable buttons when their actions are not or should not be available to users. This follows the rule of least access.

All buttons should be button tags: <button>. Buttons should have an id, and implement the class "btn btn-primary". They should contain no action attributes, such as "onclick"; however, they may contain custom data attributes (e.g., data-). All functionality associated with a button should be appended by JavaScript on page load.

Tables

Use tables only for tabular data. Use CSS columns for columnar layout, or grid elements for a grid layout - see Layout for more information. Use of tables is acceptable for aligning field labels and fields on forms. Data tables and form tables differ in their presentation. See below for more information.

All data tables should contain the following tags:

  • table
  • thead (if has header)
    • th (if has header)
  • tbody
  • tr
  • td
Required and Available table classes:
  • .table
  • .table-striped
  • .table-hover
  • .number-cell - Used to right align numbers in a table cell
  • .action-cell - Used to center buttons in a table cell

The first three classes (table, table-striped, and table-hover) should be used by default on all tables, unless otherwise specified.

Required Attributes
  • All tables must have an ID attribute.

All tables that implement the table-sort class are made sortable automatically by the system. You can make other tables sortable, by calling the $(selector).tablesorter() method, with options provided, to allow for specialized table sorting. See the section on Button Placement for an example.

Glyphs

Glyphs are appropriate to use in place of buttons to facilitate an action in locations where space is at a premium, such as table cells. Prefer to buttons to Glyphs for page actions in all other instances. The exception to this rule is standard internet memes, such as a play icon to begin the playback of rich media.

Glyphs can be used to augment system messages, by providing a visual cue. An example is using the Alert icon () paired with a sysmtem warning.

All Glyphs should have a title that explains their meaning.

Backgrounds

The web site has a default background provided by the stylesheet. Additional backgrounds may be used anywhere in the site to augment elements inside the main content area, as long as they do not interfere with or override the default background. They must also conform to the color palette, and align with the grid.

Backgrounds are the preferred method for implementing decorative or informational Glyphs, while img tags are the preferred method for implementing action Glyphs. For more information on Glyphs, see Assets.

CSS Casing

All ID elements should be TitleCase. All class elements should be hyphenated-names. All HTML elements should be lower case.

 #ExampleId{
     //Style here
 }

 .example-class{
     //Style here
 }

 div{
     //Style here
 }

Style Sections

Include style-sections in style sheets, grouping like items. Sections are demarcated by CSS comments, for example:

/*SECTION NAME*/
.css-class{
    //Styles here
}
